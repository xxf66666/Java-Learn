# Week 5 §02 · 声明式事务 `@Transactional`

> 一个注解管全套：方法成功就 commit，抛异常就 rollback。

---

## 1. 编程式 vs 声明式

**编程式（不推荐）**：
```java
PlatformTransactionManager tm = ...;
TransactionStatus status = tm.getTransaction(new DefaultTransactionDefinition());
try {
    accountDao.deduct(from, 100);
    accountDao.add(to, 100);
    tm.commit(status);
} catch (Exception e) {
    tm.rollback(status);
}
```

**声明式（推荐）**：
```java
@Transactional
public void transfer(long from, long to, BigDecimal amount) {
    accountDao.deduct(from, amount);
    accountDao.add(to, amount);
    // 抛异常自动 rollback；正常返回自动 commit
}
```

底层依然是 AOP——`@Transactional` 由切面拦截，方法前开事务、方法后提交/回滚。

---

## 2. 在 Spring Boot 里启用

Spring Boot 默认已开启，不用写额外注解。在 main 类上不需要 `@EnableTransactionManagement`，但纯 Spring 项目需要。

---

## 3. 默认行为：只回滚 RuntimeException

```java
@Transactional
public void buggy() {
    accountDao.deduct(1, 100);
    throw new IOException("...");      // ❌ checked 异常，默认不回滚！
}
```

**怎么办**：

```java
@Transactional(rollbackFor = Exception.class)    // 所有 Exception 都回滚
public void safer() throws IOException { ... }
```

**实操原则**：
- 业务异常都用 `RuntimeException`（自定义 `BusinessException`）→ 默认能回滚
- 或者 `@Transactional(rollbackFor = Exception.class)` 一刀切

---

## 4. 七种传播行为（面试高频）

事务 A 调事务 B，事务 B 该怎么办？

| 行为 | 含义 | 典型场景 |
|------|------|---------|
| `REQUIRED`（默认） | A 有事务就加入，没有就新开 | 99% 场景 |
| `REQUIRES_NEW` | 永远新开一个事务，挂起 A | 日志、审计——A 失败也不影响 |
| `NESTED` | 嵌套（保存点），A 回滚 B 也回滚，B 回滚 A 可选 | 部分业务能独立成败 |
| `SUPPORTS` | A 有事务就加入，没有就以非事务方式跑 | 偶尔事务的查询 |
| `NOT_SUPPORTED` | 永远非事务，A 的事务挂起 | 长时间查询，不想占住事务 |
| `NEVER` | A 有事务就抛异常 | 严禁事务的方法 |
| `MANDATORY` | A 没事务就抛异常 | 强制要求外部已有事务 |

### `REQUIRED` vs `REQUIRES_NEW` 直观例子

```java
@Service
class OrderService {
    @Transactional
    public void createOrder() {
        orderDao.save(...);
        try {
            logService.log("订单创建");     // 失败不要影响主事务
        } catch (Exception e) {
            // 想吞掉异常，但是 ...
        }
    }
}

@Service
class LogService {
    @Transactional(propagation = REQUIRES_NEW)
    public void log(String msg) {           // 单独的事务
        logDao.save(...);
    }
}
```

如果 `log` 用默认 `REQUIRED`，它和 `createOrder` 是同一个事务，`log` 内部抛异常即使被外面 catch 了，整个事务也会被标记 rollback-only，最终 commit 时仍然回滚。用 `REQUIRES_NEW` 独立后才能"日志失败不影响主流程"。

## 5. 隔离级别

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
```

| 级别 | 含义 | 解决 |
|------|------|------|
| `READ_UNCOMMITTED` | 能读到别人未提交的 | —— |
| `READ_COMMITTED` | 只读到已提交的 | 脏读 |
| `REPEATABLE_READ`（MySQL 默认） | 一个事务内多次读结果一致 | 脏读 + 不可重复读 |
| `SERIALIZABLE` | 串行执行 | 全部问题（但慢） |

**实操**：跟数据库默认就行（MySQL 是 REPEATABLE_READ），除非有特殊需求。

---

## 6. `readOnly` 优化

```java
@Transactional(readOnly = true)
public User findById(long id) { ... }
```

告诉数据库这是只读，能跳过一些锁机制，**纯查询方法都加上**。

## 7. 失效场景（必背）

### 7.1 自调用失效

```java
@Service
class UserService {
    public void outer() {
        this.inner();                      // ❌ this 不走代理，@Transactional 失效
    }

    @Transactional
    public void inner() { ... }
}
```

**解决**：
```java
@Service
class UserService {
    @Autowired
    private UserService self;              // 注入自己（要解决循环依赖）

    public void outer() {
        self.inner();                      // ✅ 走代理
    }
}
```

### 7.2 异常被吞

```java
@Transactional
public void buggy() {
    try {
        accountDao.deduct(1, 100);
        throw new RuntimeException();
    } catch (Exception e) {
        log.error("...", e);              // ❌ 异常被吞，不会回滚
    }
}
```

**解决**：要么重抛，要么显式标记：
```java
TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
```

### 7.3 方法非 public

`@Transactional` 在 public 方法以外的方法上**不生效**（CGLib 代理限制）。

### 7.4 没有走 Spring Bean

```java
new UserService().doSomething();          // 自己 new，不是 Spring 管的
```
